**Resumen ejecutivo**

Tomamos la decisión de simplificar: el cliente (JS en `producto/[id].astro`) hace una petición directa al endpoint de Martina usando el código canónico inyectado (`PUBLIC_MARTINA_CODE`). Para minimizar fricción y regresiones se eliminaron el `map.json` y la util `martinaMap.ts` que mapeaban `providerId` → `martinaCode`. También se eliminó la API proxy que habíamos creado temporalmente.

Resultado: menos lógica, menor I/O SSR, funcionamiento KISS para la verificación en vivo. Riesgo: algunos productos que antes resolvían por `map.json` podrían dejar de mostrar stock si el `PUBLIC_MARTINA_CODE` no cubre todos los casos; opción híbrida posible (SSR pre-fill + cliente KISS) si se requiere.

**Decisiones y justificación**

- Por petición del usuario, priorizamos KISS: ejecutar la verificación en cliente con `PUBLIC_MARTINA_CODE`.
- Eliminamos `map.json` y `martinaMap.ts` para reducir la superficie de I/O y evitar mismatches por cabeceras/origen del servicio externo.
- Mantuvimos los raw files por `code` en `data/martina_raw/*.json` (no se eliminaron) por si se quiere volver a generar `map.json` o hacer análisis offline.
- El proxy fue eliminado porque el usuario pidió explícitamente no crear una API adicional.

Riesgos:
- Podrían existir `providerId` que respondan correctamente solo con su `martinaCode` específico distinto del `PUBLIC_MARTINA_CODE`. En esos casos el cliente KISS puede mostrar falso negativo de stock.
- Si esto resulta un problema, alternativa recomendada: mantener `map.json` para SSR (pre-fill) pero que el cliente siga llamando con `PUBLIC_MARTINA_CODE` — esto combina simplicidad cliente + cobertura SSR.
